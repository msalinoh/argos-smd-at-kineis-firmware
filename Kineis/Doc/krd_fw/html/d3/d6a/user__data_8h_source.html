9 /**
10  * @page user_data_page User data Library
11  *
12  * This page is presenting the USER DATA Library.
13  *
14  * This software library handles user data. It is a central component in charge to:
15  * * get free memory and store user data coming from high layers (AT commands, sensors, ...)
16  * * provide it to several lower RF layer which will transmit it over the air (e.g. ARGOS, LoRa,
17  * NB-IOT, ...).
18  *
19  * Typically, the higher layer can be an AT command manager and the lower layer can be some network
20  * manager (NW MGR) in charge to transmit and receive data over the air. In case of hybridation,
21  * the lower layer could be a network manager which will decide on which network it is worth to
22  * transmit first.
23  *
24  * @section user_data_design_apis APIs
25  *
26  * The APIs proposed by this library are about to:
27  * * add a new message at the beginning or at the end of the FIFO
28  * * remove an element
29  * * count the number of message already located in the FIFO
30  * * count the remaining free place in FIFO
31  * * get next free memory element
32  * * get first element of the FIFO
33  * * flush the entire FIFO
34  * * Retrieve an element of the FIFO
35  *
36  * When an element is transmitted, the lower layer needs to notify the upper layers.
37  * For each message, the upper layer has to register a notification callback, so that the lower
38  * layer can call it. So far, it is assumed there is only one notification callback.
39  *
40  * On an other hand, USERDATA can notify the lower layers when som events occurs on the fifo, e.g.:
41  * * an element is added in FIFO
42  * * an element is removed from fifo
43  * * the fifo is flushed
44  * This allows lower layers to maintain their internal control code and status code.
45  *
46  * @section user_data_design_points Design/usage highlights
47  *
48  * @subsection user_data_design_point_cl Chained list
49  *
50  * Depending on the radio link quality, some message added first in FIFO may not be received first
51  * on satellite, at first transmission. Some message will need several retransmission, some other
52  * will not. Thus, some later-added messages may be removed first from the FIFO.
53  *
54  * For that purpose, the fifo is implemented as a chained list. A pointer to first element is
55  * available. Each element contains a pointer to next element. The last element of the fifo
56  * is point is pointing to NULL.
57  *
58  * As each message is independent, the FIFO is internally designed as a chained list.
59  *
60  * @subsection user_data_design_point_cs Critical sections
61  *
62  * As this library is accessed by several software entities with different priorities, some
63  * critical sections may be needed in this library for data protection against dual access.
64  * Nevertheless, it is possible to avoid too much critical sections by clearly identify and
65  * separate variables in a way to get only one writer-reader path per variable.
66  *
67  * Here is some examples:
68  * * needs_transmit:
69  * * write 0->1 by APP only
70  * * write 1->0 by APP only, after NW MGR sent notification to APP
71  * * tx_rat: write by NW MGR only
72  * * nb_transmit:
73  * * write 0->x by NW MGR only
74  * * write x->0 by RAT MGR only
75  *
76  * @subsection user_data_design_point_es Element size
77  *
78  * Depending on transmission protocol (KINEIS, NB-IOT, ...), the user data length can be different.
79  * Optimizing the memory resource with dynamic user data size in a chained list is difficult.
80  *
81  * At first step, the data buffer will be an array where each element as the maximum size allowed
82  * among all protocols.
83  *
84  * In next design iterations, two or three data size may be considered (e.g. bigger buffer for HDA4
85  * data). This will lead the higher layer to reserve place in a buffer fitting its user-data length.
86  */
